About the Optic Project

A brief recount of our inspirations for building Optic and an introduction to how it works.

Our Why

Pablo Picasso famously said that “everything you can imagine is real”. We have set out on a mission to make that literally true in software development. There are so many good ideas that never see the light of day because they are too challenging to make real — this hampers progress, and it’s everyone’s loss. A fusing of the best creative minds with innovative tools has always been a potent mixture for progress and its our job to create tools that help creators bring their imaginations to life.

Human Machine Collaboration

Making programming more intuitive has been a long stated goal of many developers. In the past this problem was approached by building new libraries, languages, visual tools, and other abstractions for programmers to learn. Better abstractions certainly will continue to drive progress but so will making it easier to use and glue together the ones we already have. This is where we see a huge opportunity to apply automation to the discipline of programming.

As it stands today, programming knowledge is diffused in a variety of places. There are tutorials, that example your co-worker emailed you, videos, wikis, Q&A sites, docs pages and troves of raw source code sitting on Github. The landscape in many ways resembles the internet as a whole before Google. This abundance of resources is great; however, it takes a human to find, interpret and act on this information and human time is scarce and expensive.

After spending over 100 hours this year watching people code and realized that process, not coding, was where they spent most of their time and experienced their greatest frustrations.

The primary goal of our project is to organize the world’s programming knowledge and make it accessible through automation. While working towards this objective we tried to imagine how future developers might collaborate with computers while programming. Following dozens of user interviews, and many design sessions we began to focus on the following questions: 

  • What if there were a way to have programmers and machines collaborate on the same codebase without getting in each other’s way?
  • What if you could ask documentation questions? What if it could do things for you?
  • What if there you could import conventions and best practices the same way you import code? That could be in the form of knowledge from teammates, a developer you look up to, or the team behind a library you want to use.

We believe that when programmers should be able to elevate themselves to the most important problems and delegate repetitive/low-value work to a computer. In our experience this leads to happier developers, fewer bugs, faster cycle times, and higher quality. That all sounds great on paper, but figuring out how to build a technology and developer experience that achieves these ends without stymieing control or quality has been an open challenge for years. To say we have fully solved it would be dishonest. We’ve made enough progress to add value to end users, and with the support of our users and the community, we trust we will make more.

So what are you actually releasing? Simple, one product and one technology.

Optic Agent —  clippy for code

We built the Optic Agent to help programmers automate work, share knowledge and enforce conventions. Once installed, the Agent will monitor your code and function as your personal programming assistant. We believe you should not have to change all of your tools or abandon your battle hardened workflow to benefit from a little automation. So our key design focus while developing the agent was to make it discreet and impartial.

The Agent can make use of your team’s knowledge or you can import knowledge from individual developers and companies you trust. Once configured, this knowledge can be used by the Agent in several ways:

  • Perform searches — just start typing questions in your editor and Optic will pull up results from its knowledge-base. Once a result is selected and configured code gets generated into your project. ie “Process Credit Card”, “Send Text Message” or “Issue JWT Token”. We’re working with NLP experts to push the limits on what you can ask Optic to generate. Supporting compound questions like “Create a route that lists all of a user’s friends” is a stretch goal and one we’ll be able to move towards as our ecosystem grows larger. 
  • Make Suggestions — since the Agent knows a lot more about your code and how different pieces may fit together than any other IDE, it can suggest the next things you might want to do based on your context. For instance if you have a Mongoose Model selected, Optic will ask if you want it to generate CRUD routes & queries for that model and help your configure them.  
  • Connect Code — A major headache for programmers is keeping their code synchronized across projects and teams. A simple example would be keeping your models synced between your server and client code. Optic can store implicit connections between different parts of your code (even across repos) and keep everything consistent with whatever you define as the ground truth.

Unlike other code generators Optic does not break if you change something it generates. We actually encourage you to make changes by hand whenever you need to and have designed the tool to work with them gracefully. Without getting too technical this works because there is no hidden information in an Optic Project; the code itself is always the ground truth. Change it as you see fit. Collaboration is not a one way street.

We’ll be releasing videos over the next few weeks leading up to release showcasing the Agent and how our early users are making use of it.

Sourcegear — open tools for automated programming

For machines and humans to collaborate on code they both need to understand that code and have a standard way to interface with it. We have our brains, fingers and keyboard and those work quite well. Computers have a more difficult time with this because reading and generating code is a hard problem. The state of the art right now in code generation makes no accommodations for collaboration between machines and humans since every time code is generated it overwrites any changes a human might have made. 

We’re leveling the playing field with Sourcegear: Optic’s toolchain for automated programming. Sourcegear greatly reduces the complexity in building automated programming tools by abstracting away syntax, formatting and structure. Without it the Optic Agent would not be possible. 

Sourcegear exposes a RESTful API on your local machines and allows you to query for different entities within your source code. Want to find all the HTTP Routes in your project, run at GET request. Want to insert something new, use a POST request. Updating something? Use PUT. 

The API exchanges JSON summaries of your code. The summary of an HTTP Route from the API might look like:

{url: ‘/resource/create’, method: ‘post’, parameters: [ {in: ‘query’, name: ‘test’, rule: ‘email’}]}

If you change something in that model and PUT it back to the API, it’ll update your code, wherever it is, to conform to the updated summary. This sort granular control over source code is a hard problem. It’s only been with the aid of machine learning techniques that we’ve been able to make our approach reliable enough that it isn’t even noticed.

Sourcegear’s abstraction dramatically reduces the complexity of any tool that seeks to write, read or otherwise manipulate source code. We hope this platform will usher in a new kind of programming tool where computers and humans work on code together and each masters the tasks that best suite them. 

A full explanation of how Sourcegear works will be released along with its source code in January ’18.

Open Source

Optic is our attempt to bring these concepts to life. We are doing this not for ourselves, and not only for our users, but rather for the benefit of the community as a whole. If we are being serious about trying to influence the future of programming, and we are, we must also be serious about the way in which we do it. We do not believe one group should control the means of production for software and we think it’s much better to do our work out in the open for all to see, comment on, contribute to and use to solve real world problems.
